The problem

Three failure modes traditional security can't see

WAFs and API gateways inspect HTTP shape. They have no concept of what an autonomous agent is trying to do, or whether it should be doing it right now.

Runaway loops

The financial killer

A stuck agent can call your LLM API recursively thousands of times in an evening, turning a logic bug into a five-figure invoice before anyone notices.

Indirect injection

The security nightmare

Untrusted data — an email, a webpage, a ticket — can carry hidden instructions. The model can't tell "system prompt" from "thing I just read," and executes anyway.

Destructive execution

The blast radius

Agents now hold write access — files, containers, databases. One bad tool call argument and there's no undo button.

How it works

One proxy, three checkpoints

AgentGuard sits in the path of every tool call your agent makes — not after the fact, in line, before execution.

interceptor

main.go

Authenticates the agent session, decodes the JSON-RPC tool payload, and routes it through the breaker and scanner before anything executes.

tool: "delete_user"
args: {id: 19}
→ forwarded for inspection
circuit breaker

breaker.go

Tracks call frequency and semantic similarity in a rolling window. More than 10 near-identical calls in 30 seconds trips the breaker and freezes the session.

window: 30s
threshold: 10 calls
similarity: >90%
sandbox

sandbox.go

Routes write operations into an isolated namespace with Landlock filesystem restrictions. If the agent tries to leave its directory, the kernel blocks it.

scope: /tmp/session-id/
landlock: enforced
escape attempts: 0 allowed
